**Tooling note (not a site defect):** a `preview_screenshot` capture of the
mobile home page rendered the hero visual as a narrow ~208px-wide column
with a large dark void filling the rest of the 375×812 frame. DOM inspection
(`getBoundingClientRect`, computed styles, and `elementFromPoint` sampled
across the full viewport width) all confirm the actual rendered layout is
correct, full-bleed 375×812 with no transform/clip applied — `object-fit:
cover` on a `375×812` box. `devicePixelRatio` on this session is 2, which is
the likely source of a capture/downscale mismatch in the screenshot tool
itself. Per the environment's own guidance ("verify STATE via classes/DOM,
never via waiting for opacity transitions" / screenshots are unreliable for
precise verification), this is recorded as a harness artifact, not a build
issue — but worth a human glance at the real device/browser if there's any
doubt.
